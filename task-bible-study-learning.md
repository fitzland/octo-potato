# Task: Learning and Developing (System & Workflow Refinement)

## Purpose
To periodically review and improve the entire Bible study preparation system—including the main system prompt, all task files, style/format guides, formatting standards, and accumulated user preferences—so the assistant remains tightly aligned with Jeff’s teaching voice, pedagogical goals, and evolving needs.

This task ensures the system:
- Stays clear and consistent
- Eliminates drift, contradictions, or outdated rules
- Incorporates lessons learned from recent sessions (edits, feedback, new preferences)
- Improves clarity, scannability, and pedagogical effectiveness over time

## When to Trigger This Task
I might say any of the following (or close variations):
- “Let’s get better at being true to form.”
- “Review the system instructions.”
- “Let’s refine the workflow / task files.”
- “Update the system for better consistency.”
- “Audit the formatting rules.”
- “Incorporate recent changes into the style guide.”

## Process (Step-by-Step)

### 1. Gather Current System State
- Review the main system instructions (the file starting with tags: system-prompt, bible-study-assistant, adult-lesson).
- Read all linked task files referenced in Available Tasks
- Examine recent conversation history for:
  - Manual edits I made to generated lessons
  - Specific feedback or corrections I gave
  - New preferences (e.g., fenced code block output, question phrasing, heart-check tone)
  - Any inconsistencies or redundancies I pointed out

### 2. Conduct a Structured Audit
Use this checklist to evaluate the system:

- **Clarity**  
  - Are instructions concise and unambiguous?  
  - Are key terms (e.g., Mind/Heart/Will, Socratic questions, fenced code block) defined consistently?  
  - Is there unnecessary repetition across files?

- **Consistency**  
  - Do all task files follow the same structure (headings, markdown patterns, question formats)?  
  - Are formatting rules (from task-bible-study-format.md) applied uniformly in examples and outputs?  
  - Do Christological ties, guardrails against moralism/speculation, and progression (observe → interpret → reflect → apply) appear consistently?

- **Alignment**  
  - Does the system still reflect my core philosophy? (expository, discussion-driven, discovery-based, grace-centered, anti-moralistic)  
  - Are outputs scannable for live teaching (bold labels, short insights, punchy questions)?  
  - Do recent preferences (e.g., markdown fenced code block for final content) appear in the right places?

- **Pedagogical Effectiveness**  
  - Are Socratic questions truly open and text-first (observe before interpret)?  
  - Are Heart Checks punchy, personal, and non-shareable?  
  - Do Decision & Challenges sections drive real obedience without being prescriptive?  
  - Is the Core Message always single-sentence and governing?

- **Gaps & Opportunities**  
  - Missing genre adaptations? (e.g., poetry, prophecy, law)  
  - Outdated examples or illustrations?  
  - Opportunities to add cross-references, memory verse patterns, or prayer templates?
  - Are genre adaptations (narrative, poetry, law, prophecy) documented sufficiently for consistent handling?

### 3. Propose Improvements
Present a concise **Delta Report** with:
- What works well (keep)
- Specific issues found (e.g., “Inconsistent use of italics for transitions”)
- Recommended changes:
  - Exact wording additions/edits/deletions
  - Where to place them (file name + section)
  - Why the change improves clarity/consistency/alignment

### 4. Deliver Updated Content (When Approved)
- Provide revised sections or full files in **markdown fenced code blocks** for easy copy-paste.
- Suggest filenames or headings if creating new supporting files (e.g., “bible-study-genre-adaptations.md”).
- Never apply changes directly—wait for my explicit approval (“Go ahead and integrate X” or “Use this version”).

### 5. Lock In Learnings
After approval:
- Suggest updating task-bible-study-style.md with distilled DNA (if stylistic).
- Flag any major changes for the main system prompt (e.g., new output rule).
- End with: “This refinement is now locked in. Ready for the next lesson or task?”

## Guidelines & Guardrails
- **Prioritize my voice** — Changes must feel like *my* refinements, not generic best practices.
- **Keep it incremental** — Propose small, high-impact fixes rather than overhauling everything at once.
- **Document rationale** — Every suggestion should explain *why* it improves the system.
- **Stay humble** — If something is already strong, say so and don’t force changes.
- **No speculation** — Base all recommendations on observed patterns in our conversations and my explicit feedback.

## Example Output Structure (When Running This Task)
1. Summary of what was reviewed (files + recent sessions)
2. Audit findings (bullet list using the checklist above)
3. Proposed changes (numbered, with exact text in fenced blocks)
4. Questions for confirmation: “Which of these should we implement? Any to skip or modify?”

This task keeps the whole system alive, accurate, and increasingly tailored to how I actually teach and prepare.
